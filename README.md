# Архитектура системы генерации ответов на обращения

## 1. Стек технологий

**Инфраструктура и API:**
* **FastAPI:** Асинхронный фреймворк для оборачивания ML-моделей и построения API.
* **PostgreSQL:** Хранение метаданных обращений, статусов обработки и итоговых текстов.

**Обработка входящих данных (OCR и NLP):**
* **EasyOCR / Tesseract:** Извлечение текста из приложенных скан-копий.
* **RuBERT:** Входная классификация обращений по тематикам (например, жалобы, запросы справок).
* **Natasha / Yargy:** Rule-based извлечение именованных сущностей (NER) — ФИО, адреса, даты, названия организаций.

**Модуль RAG и Генерация:**
* **LangChain:** Оркестрация пайплайна. Связывает компоненты в единую цепочку: прием запроса -> поиск контекста -> генерация.
* **Ollama:** Платформа для локального запуска квантованных LLM. Локальный инференс необходим для защиты персональных данных заявителей.
* **Qdrant / FAISS:** Векторная база данных для хранения эмбеддингов нормативно-правовых актов (НПА) и регламентов.
* **BGE-m3 / rubert-tiny2:** Embedding-модели для перевода текстов в векторное представление.

## 2. Пайплайн обработки (Верхнеуровневая архитектура)

```text
[Источник: Email / Веб-форма / API]
       │
       ▼
1️⃣ ПРЕПРОЦЕССИНГ И OCR
   ├─ Проверка формата.
   ├─ Скан -> [EasyOCR] -> Текст
   └─ Очистка текста.
       │
       ▼
2️⃣ NLP: КЛАССИФИКАЦИЯ И NER
   ├─ [RuBERT] -> Определение тематики.
   └─ [Natasha] -> Извлечение ФИО, адресов -> (Сохранение в PostgreSQL)
       │
       ▼
3️⃣ RAG: ПОИСК КОНТЕКСТА
   ├─ Векторизация обращения.
   └─ [Qdrant / FAISS] -> Извлечение релевантных статей НПА.
       │
       ▼
4️⃣ ГЕНЕРАЦИЯ (LLM)
   ├─ [LangChain] -> Формирование промпта с контекстом.
   └─ [Ollama] -> Генерация черновика ответа.
       │
       ▼
5️⃣ ПОСТПРОЦЕССИНГ
   ├─ Вставка NER-сущностей в официальный шаблон.
   └─ Выгрузка готового документа / API ответ.
